## Misunderstanding Memo

* Nested resource URLs are common in REST-style API design.
```http
GET /users/{userId}/orders
GET /customers/{customerId}/invoices
```

* Use `async` for database access in Web APIs.
  Database calls involve I/O waiting. Using `async/await` prevents the request thread from being blocked while wating for the database.

* `Include + FirstOrDefaultAsync` can still load multiple child records.
  `FirstOrDefaultAsync` returns one `User`, but that `User` can contain many `Orders`.
```csharp
var user = await _db.Users
    .Include(u => u.Orders)
    .FirstOrDefaultAsync(u => u.Id == userId);
```

```csharp
user = User {
    Id = 1,
    Name = "Taro",
    Orders = [
        Order { Id = 101 },
        Order { Id = 102 },
        Order { Id = 103 }
    ]
}
```

* `new{...}` creates an anonymous type in C#.
  this is valid C# syntax.
```csharp
var result = user.Orders.Select(o => new
{
    o.Id,
    o.TotalPrice,
    o.CreatedAt
});
```

* Do not make route parameters nullable just to handle invalid input manually.
int userId itself cannot be null. If a non-integer value such as "abc" is passed, the problem is not a null check issue but a model binding / validation issue. With [ApiController], ASP.NET Core automatically returns 400 Bad Request when model binding fails, so you usually do not need to make the parameter int? and check it manually.

```csharp
[ApiController]
[Route("users")]
public class UsersController : ControllerBase
{
    [HttpGet("{userId}/orders")]
    public IActionResult GetUserOrders(int userId)
    {
        return Ok();
    }
}
```

Without [ApiController], you may need to check ModelState.IsValid manually.
```csharp
if (!ModelState.IsValid)
{
    return BadRequest(ModelState);
}
```

* Route constraints can make the route accept only integers.
  If {userId:int} is used and "abc" is passed, the route does not match, so the result is usually 404 Not Found.

```csharp
[HttpGet("{userId:int}/orders")]
public IActionResult GetUserOrders(int userId)
{
    return Ok();
}
```

Difference between model binding failure and route constraint failure:
```
/users/{userId}/orders      + "abc" → route matches, int binding fails → 400
/users/{userId:int}/orders  + "abc" → route does not match → 404
```
