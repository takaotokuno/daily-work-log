# キャリア振り返り：技術経験の棚卸し

## 対象テーマ

T3 Stackによる在庫管理アプリ開発

## 実際に自分でやったこと

* Webアプリケーション開発の実務経験がほぼない状態から、学習兼ポートフォリオとして在庫管理アプリの実装に取り組んだ。
* 短期間で形にする必要があったため、Next.js / TypeScript / Prisma / tRPC / NextAuth などが含まれる T3 Stack を選択した。
* 環境構築から始め、PrismaによるDB定義、画面作成、フォーム入力、保存処理まで実装した。
* shadcn/ui を利用して、最低限アプリケーションらしく見えるUIを作成した。
* 保存ボタンを押すと、入力内容がDBに登録されるところまでは実装できた。
* 実装中に、フロントエンド、API、DB、ORM、認証、Dockerなど、Webアプリ開発に必要な要素の全体像に触れた。

## まだ理解が浅いこと

* Next.js の App Router、Server Components、Client Components の責務分担。
* 認証・認可まわりの仕組み。特に NextAuth がなぜ動いているのか、セッション管理がどう成立しているのかは理解が浅い。
* TypeScript の型設計。型を安全に使って設計するというより、動くコードを追いながら書いていた段階。
* Docker についても、コンテナ・ボリューム・ネットワークの基礎は触れたが、設計判断できるほどではない。

## 面接で言うと危ない表現

* T3 Stackを使ってWebアプリを一通り開発できます。
* Next.js、tRPC、Prisma、NextAuthを実務レベルで扱えます。
* 認証付きの在庫管理アプリを設計・実装しました。
* Dockerを使った開発環境を自分で設計できます。
* TypeScriptでフルスタック開発ができます。

## 面接で安全に言える表現

* Webアプリ開発未経験に近い状態から、短期間でT3 Stackを使った在庫管理アプリの実装に取り組みました。
* 技術を深く選定できる段階ではなかったため、必要な要素が一通り含まれているT3 Stackを選び、環境構築からDB保存処理まで実装しました。
* 完成度は高くありませんが、フォーム入力からDB登録までの一連の流れを自分で確認できるところまでは作りました。
* その経験を通じて、フロントエンド、API、ORM、DB、認証、DockerがWebアプリの中でどう関係するのか、全体像を掴むきっかけになりました。
* 現時点では、T3 Stackを実務レベルで使えるとは言えませんが、ASP.NETでの業務経験と照らし合わせながら、Webアプリ全体の設計を理解し直している段階です。
* 今後は、TypeScriptとNext.jsの基礎を補強し、同じ在庫管理アプリをもう一度作り直すことで、理解を実装に結びつけたいと考えています。

## 次に補強すべきこと

* まずは認証なしのシンプルなCRUDを、Next.js / TypeScript / Prisma で作り直す。
* その後、ログイン機能、ユーザーごとのデータ分離、認可チェックを追加する。
* ASP.NETで理解しているレイヤー分割やAPI設計と、Next.js / T3 Stack の構成を対応づけて整理する。
* TypeScriptの型、Zodによるバリデーション、API境界での型安全性を重点的に学ぶ。
* 面接で話す場合は、「T3 Stackを使いこなせる」ではなく、「未経験領域を短期間でキャッチアップし、最低限動くものまで到達した経験」として説明する。


## 対象テーマ

Unityゲーム開発

## 実際に自分でやったこと

* プログラミング学習の一環として、Unityで簡単なボードゲーム開発に取り組んだ。
* 最初は小さく作り始めたが、人間プレイヤーとCPUの切り替え、アンドゥリドゥ、先手・後手の管理、スキル・ステータス追加など、機能を増やすたびに既存の設計が崩れやすいことを経験した。
* 拡張のたびに全体へ影響が出る原因を考え、GoFのデザインパターン、クリーンアーキテクチャ、DI、DDDなどの設計手法・考え方を学びながら、3〜4回ほど設計から作り直した。
* ゲームの状態、プレイヤー、ターン、入力、スキル、CPU処理などをどう分離すべきかを考え、単に動くコードではなく、変更しやすい構造を意識するようになった。
* AIも活用したが、自分の設計理解が浅い状態で生成されたコードをそのまま使っても、保守しやすいものにはならないことを体感した。
* その経験を通じて、AIを使う場合でも、設計方針・責務分担・修正方針を人間側が判断できる必要があると学んだ。

## まだ理解が浅いこと

* Unity特有のライフサイクル、MonoBehaviour、Prefab、Scene管理を前提にした設計の作り方。
* DDDやDIをUnityにどこまで適用すべきか、逆にどこからが過剰設計になるかの判断。
* CPUの探索処理や評価関数の設計、探索高速化。
* UI設計や演出面。現状は機能中心で、見た目や操作感はまだ無機質。
* 認証、DB保存、クラウド連携など、Unity単体の外側にある仕組みとの接続。
* Webブラウザゲームやスマホゲームとして公開するためのビルド、配信、運用まわり。

## 面接で言うと危ない表現

* Unityで本格的なゲーム開発ができます。
* DDDやDIをUnityで実践できます。
* CPU対戦機能を高度に実装しました。
* 設計パターンを使って拡張性の高いゲームを作りました。
* AIを活用して効率的にゲーム開発しました。

## 面接で安全に言える表現

* 学習目的でUnityを使い、ボードゲーム開発に取り組みました。
* 機能追加を重ねる中で、プレイヤー切り替え、CPU処理、ターン管理、スキル管理などが密結合になりやすく、拡張のたびに全体が壊れやすいことを経験しました。
* その問題を解消するために、GoFのデザインパターン、DI、DDDなどを学びながら、設計を何度か作り直しました。
* 完成度の高いゲームを作ったというより、変更に弱いコードをどう分離し、責務を整理するかを学んだ経験です。
* 特に、ゲーム開発を通じて「状態管理」「入力と処理の分離」「ルールとUIの分離」「拡張時に壊れにくい構造」の重要性を実感しました。
* AIも利用しましたが、自分の理解が浅い状態では良い設計にはならず、AIを使う側にも設計判断力が必要だと学びました。

## 次に補強すべきこと

* 現在のゲームを、状態管理・入力処理・ルール判定・UI表示の責務に分けて再整理する。
* CPUの探索処理や評価関数を改善し、CPUの強さを段階的に調整できるようにする。
* UIを改善し、操作感や視覚的なわかりやすさを高める。
* 認証、DB保存、クラウド連携を追加し、Unity単体ではなく、Webアプリやスマホアプリに近い構成へ発展させる。
* 面接で話す場合は、「Unityでゲームを作れる」ではなく、「機能追加で壊れやすくなるコードを経験し、設計改善に取り組んだ」と説明する。
